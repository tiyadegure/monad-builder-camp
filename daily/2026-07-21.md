### 1. 项目选择：为什么做 CLI 质押 Agent

Day 15 的 Builder Profile 已经明确：我能给团队的 Dev 侧贡献是\"把链上协议封装成 Agent 可调用的能力\"。PR #104 解决了 adapter 层（函数签名 / selector / Receipt parser），但 adapter 本身只能被开发者直接调用，没有用户界面。

Week 3 Dev 的最小闭环是：
- 用户说自然语言意图（\"Stake 1 MON into aPriori\"）
- Agent 通过 Moss `discover → load → action` 构建正确交易
- 交易草稿交用户签名广播（不碰私钥）

`stake-agent.mjs` 就是这个闭环的最简原型。

### 2. `stake-agent.mjs` 实现要点

- **入口**：`node stake-agent.mjs <金额MON> [receiver地址]`，默认 receiver 为占位账户
- **运行时**：`system.monadRuntime({ rpcUrl: "https://rpc.monad.xyz" })` 直连 Monad 主网
- **交易构建**：`registry.action("apriori", "stake", ACCOUNT, { amount, receiver })`，Moss 内部映射到 `deposit(uint256, address)` + selector `0x6e553f65` + payable
- **Receipt 估算**：真实主网 `convertToShares` 视图调用，把 MON 金额换算成预计 aprMON 数量
- **退出提示**：unstake(requestRedeem) → claim(redeem) 两步异步流程

### 关键认知

- **adapter 解决了\"能不能调\"，原型解决\"好不好用\"。** PR #104 的 adapter 只暴露函数签名和地址，`stake-agent.mjs` 把它包装成用户一句话能触发的结果。
- **Moss 的\"只模拟不签名\"是安全特性，不是缺憾。** 原型不需要广播，交易草稿正确就已经是可检查的 Evidence。
- **Week 3 的 Dev 项目不必等组队再开始。** 哪怕最后是单干，可运行的 CLI + 测试截图 + README 也是 Hackathon 能交的货。
- ** Receipt 估算用真实 `convertToShares` 而不用 mock 数字。** 这个决定让原型从\"代码能跑\"升级成\"链上数据可验证\"。
