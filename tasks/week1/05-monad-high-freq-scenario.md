## Research：为什么 Monad 更适合 Onchain Task Board

### 1. 这个产品为什么需要频繁交互

Onchain Task Board 是一个链上悬赏任务系统：任何人发布任务（带赏金），其他人认领、提交、获得验证和奖励。每个生命周期节点——发布、认领、提交、评审、发放——都是一次链上状态变更。如果再加上 AI Agent 自动认领和提交任务，交互频率会更高：一个活跃的任务板每天可能产生几十到上百次状态更新。

### 2. 链很慢或手续费高时，用户体验会怎样

在以太坊主网上，每次操作（认领、提交、评审）都要等 12 秒出块 + 几美元 gas。用户发布一个 5 美元赏金的任务，gas 可能就要 3 美元，完全不经济。更关键的是"等待感"——点击"认领任务"后等 15 秒才确认，这种延迟在任务系统里会让用户觉得产品是坏的。如果用 L2，虽然便宜了，但跨链提款延迟和流动性碎片化又带来新问题。

### 3. Monad 能带来什么帮助

Monad 的 10,000 TPS 和亚秒级出块意味着：发布任务、认领、提交这些操作几乎是即时确认的。Gas 费用极低，小额悬赏（1-5 美元）变得可行。EVM 兼容性让现有 Solidity 合约和工具链（Foundry、Ethers.js）直接可用，不需要重写。这让"链上任务系统"从一个理论上好但实践中太贵太慢的想法，变成一个真正可用的产品。

### 4. 为什么需要链上而不是普通数据库

链上任务板的核心价值是**信任和可组合性**。赏金锁定在合约里，完成就自动发放，不存在"平台不给钱"的问题。任务完成记录是公开的、可验证的——任何人都能查到某个地址完成了哪些任务、获得了多少奖励。这本身就是 Builder 的链上简历。而且合约是可组合的：其他协议可以直接读取你的任务完成记录，自动给予信用额度或权限。这些是数据库做不到的。

## Tech：高频交互 Demo 功能清单

### 核心合约

| 功能 | 说明 | 交互类型 |
|------|------|----------|
| `createTask(title, reward)` | 发布任务，锁定 MON 赏金 | write |
| `claimTask(taskId)` | 认领任务，锁定给认领者 | write |
| `submitProof(taskId, proof)` | 提交完成证明 | write |
| `approveTask(taskId)` | 验证通过，自动发放赏金 | write |
| `cancelTask(taskId)` | 取消任务，退回赏金 | write |
| `getTask(taskId)` | 读取任务详情 | read |
| `getActiveTasks()` | 读取所有进行中的任务 | read |
| `getMyTasks(address)` | 读取某人的任务历史 | read |

### 高频交互场景

1. **实时任务列表** — 新任务发布后通过事件监听即时显示，无需刷新
2. **一键认领** — 点击即上链，亚秒确认，体验接近 Web2 按钮
3. **批量操作** — 一次交易认领多个任务，或同时提交多个证明
4. **AI Agent 自动认领** — Agent 监听新任务事件，匹配技能后自动 claim
5. **链上排行榜** — 根据完成任务数和获得赏金自动排名，无需后端

### 为什么这些在 Monad 上体验更好

- 认领/提交等 write 操作 < 1 秒确认，用户无感知延迟
- Gas 低到可以做 1 美元级别的微赏金
- 10,000 TPS 支撑大量并发任务操作，不会拥堵
- EVM 兼容，直接用 Foundry 部署，用 Ethers.js 前端交互
