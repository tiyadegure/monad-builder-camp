## 背景问题

Monad 的**活跃验证者集合（active validator set）上限被写死为 200**。这意味着无论有多少人质押、多少个节点想参与共识，同时参与出块和投票的验证者最多只有 200 个。

问题：上限 200 限制了网络能容纳的验证者数量，直接限制了去中心化程度和容错能力。随着 Monad 生态增长，更多节点想参与共识却被挡在门外。

---

## 核心方案

把 `ACTIVE_VALSET_SIZE` 这个参数从 **200 提高到 300**。

就这么简单——改一个硬编码的数字。但 MIP 里说明了为什么是 +100 而不是更多或更少：
- +100 是约 50% 的扩容，有意义但不激进
- 更大的步长（比如直接到 500）风险高，可能导致 MonadBFT 投票轮次的消息复杂度突然恶化，性能不可预期
- 更小的步长（比如 +20）流程成本高但收益不成比例

选择机制不变：谁进活跃集合仍按质押量排序，只是池子从 200 扩大到 300。

---

## 影响对象

| 对象 | 影响 |
|------|------|
| 验证者（validator） | 更多节点有机会进入活跃集合，参与出块和投票 |
| 质押用户 | 可以选择更多验证者，去中心化增强 → 网络更安全 |
| 共识层（MonadBFT） | 每轮共识消息量增加（200→300 个参与者） |
| 执行层 | 不受影响，除非链上合约读取了验证者集合，这类合约需要检查兼容性 |
| 普通用户 / DApp | 间接受益：网络容错性和经济安全性更强 |

---

## 关键术语

| 术语 | 解释 |
|------|------|
| **Active Validator Set** | 当前周期内实际参与共识（出块+投票）的验证者集合 |
| **ACTIVE_VALSET_SIZE** | 活跃验证者集合的上限参数，硬编码为 200 |
| **MonadBFT** | Monad 的共识机制，用 fan-out/fan-in 模式降低通信开销 |
| **Message Complexity** | 共识轮次中消息交换的数量，验证者越多消息越多 |
| **Fault Tolerance** | 容错能力——系统能承受多少个节点作恶或宕机仍正常运行 |
| **Economic Security** | 经济安全性——攻击网络需要的资金成本，验证者越多成本越高 |
| **Standards Track / Core** | MIP 分类，Core 表示影响共识或执行核心层 |

## 争议与风险

**提案自身提到的风险：**
1. **共识可扩展性**：活跃集合变大 → 每轮消息数增加 → 共识开销上升。这是 +100 而非更大的原因
2. **Sybil 风险**：作者认为不变（选择机制仍按质押量），不需要额外缓解

**我的质疑（reading 后仍未完全确认）：**
- MIP 说"不影响执行层"，但如果大量 DeFi 协议依赖验证者集合做_slashing 或_委托逻辑，兼容性审查谁来做？
- "更大步长风险高"是定性判断，没有给性能测试数据支撑 +100 是安全上限
- 状态是 Draft，还没有量化模拟（如 300 验证者下的实际 TPS / 最终性延迟）

---

## 产品启发

1. **去中心化是产品卖点**：Monad 主打"高性能 + 去中心化不妥协"，MIP-9 是这句话的工程落地——用架构优势（MonadBFT 线性通信）支撑更大验证者集合
2. **治理体系可观察**：作为 Research，MIP 论坛是判断"项目是否真的在认真做去中心化"的一手信息源，比看营销文案可靠
3. **参数化思维**：好的协议设计把"治理可调参数"和"硬编码常量"分开，MIP-9 暴露了 Monad 目前还是硬编码（需硬分叉改 200→300），说明这块还没完全参数化

---

## 我的疑问（研究视角）

1. **MIP-9 反映的是"供给侧扩容"还是"需求侧压力"？** 如果当前 200 个位置远未满，这个 MIP 更像是为未来增长提前铺路，而非解决当下瓶颈。对研究者来说，判断一个 MIP 是"主动规划"还是"被动救火"，能区分团队是在引领还是补救。

2. **Monad 的"去中心化"叙事和参数现实之间有多大差距？** 官网强调"消费级硬件 + 大验证者集合"，但活跃集合仍靠硬编码上限管控。提高上限是朝叙事靠近的一步，但 Core 类改动需要硬分叉，说明"去中心化"目前还不是完全链上治理自动调节的。

3. **这类"改一个数字"的 MIP 在 Monad 治理里占比多少？** 如果大量 MIP 都是 Core 硬编码参数调整，说明协议成熟度还早，治理更多在"调参"而非"架构演进"——这对评估生态阶段有信号意义。

---

## 一句话总结

> MIP-9 把一个数字从 200 改成 300，背后是 Monad "高性能不牺牲去中心化" 路线的具体体现——但作为 Draft，它还需要性能数据来证明 +100 是安全边界，也折射出早期公链治理仍以"核心团队提案 + 参数硬编码"为主的特征。
