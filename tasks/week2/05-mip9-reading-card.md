## 背景问题

Monad 的**活跃验证者集合（active validator set）上限被硬编码为 200**（`ACTIVE_VALSET_SIZE = 200`）。

在 Monad 的共识设计中，每个 epoch 周期内，只有进入"活跃集合"的验证者才有权参与出块（block proposal）和投票（voting）。共识守护进程（consensus daemon）把 `ACTIVE_VALSET_SIZE` 当作**上限**来执行：无论全网有多少节点在质押、多少节点想参与，同时活跃的验证者最多 200 个。

**这带来两个问题：**
1. **去中心化天花板**：验证者数量被 200 卡死，意味着网络的容错能力和经济安全性存在一个硬上限。想参与的节点再多，也进不了活跃集合。
2. **增长瓶颈**：随着 Monad 主网上线后生态扩张，质押者增多，但活跃验证者数不增长 → 新验证者被挡在门外，网络控制权集中化风险上升。

MIP-9 要解决的，就是把这个天花板从 200 抬高。

---

## 核心方案

**把 `ACTIVE_VALSET_SIZE` 从 200 提高到 300。**

这是一个单参数修改，但提案花了相当篇幅论证"为什么是 +100，不是更多或更少"：

- **+100 ≈ 50% 扩容**：有意义但不激进，能在不剧烈改变共识层负载的前提下扩大参与
- **更大的步长（如直接到 500）风险高**：MonadBFT 每轮投票的消息复杂度会随验证者数上升，步长太大可能导致性能不可预期的退化
- **更小的步长（如 +20）不划算**：流程成本（走一次 MIP + 硬分叉）和收益不成比例

**关键不变项：**
- 选择机制不变——谁进活跃集合仍按质押量（stake）排序，只是池子从 200 扩大到 300
- 执行层（execution pipeline）不受影响，除非有链上合约读取验证者集合，这类合约需自行检查兼容性

**激活方式：** 需要把硬编码的 `ACTIVE_VALSET_SIZE = 200` 改成 `300`，属于硬分叉类改动（Core 类 MIP）。

| 对象 | 影响 | 方向 |
|------|------|------|

| 执行层 / DApp | 基本不受影响；若合约读取验证者集合需审查兼容性 | ➖ 中性偏审查 |
| 普通用户 | 间接受益：容错性、经济安全性更强，单点控制风险降低 | ✅ 更安全 |
| 协议开发者 | 若维护读取验证者集合的合约，需做兼容性检查 | ⚠️ 额外工作 |

## 关键术语

| 术语 | 解释（用自己的话） |
|------|------|
| **Active Validator Set** | 当前 epoch 内实际参与共识（出块 + 投票）的验证者名单，不是所有质押者，而是"上岗"的那批 |
| **ACTIVE_VALSET_SIZE** | 活跃集合的人数上限，目前硬编码 200，提案要改成 300 |
| **MonadBFT** | Monad 的共识算法，用 fan-out/fan-in（先分发再汇总）模式，让通信开销随验证者数**线性**增长而非指数膨胀 |
| **Message Complexity** | 一轮共识里节点之间发了多少消息。验证者越多，消息越多，这是扩容的主要成本 |
| **Fault Tolerance** | 容错——系统能扛住几个节点作恶/宕机还正常运行。验证者越多，容错绝对值越高 |
| **Economic Security** | 经济安全——攻击网络要控制的质押资金门槛。验证者越多且分散，攻击成本越高 |
| **Stake-weighted Selection** | 按质押量排序选谁进活跃集合，提案明说这个机制不变 |
| **Standards Track / Core** | MIP 分类。Core = 影响共识或执行核心层，需要硬分叉激活 |
| **Backwards Compatibility** | 向后兼容。提案称本次改动是"additive"（增量式），现有活跃验证者不受影响 |

## 争议与风险

### 提案自身承认的风险
1. **共识可扩展性**：活跃集合变大 → 每轮消息数增加 → 共识开销上升。这是提案选择 +100 而非更大步长的核心原因。
2. **Sybil 风险**：作者判断不变——选择机制仍按质押量，扩大集合不降低作恶门槛，无需额外缓解。

### 我的质疑（读原文后仍未确认）
1. **兼容性审查责任不清**：提案说"执行层不受影响，除非合约读取验证者集合"。但 Monad 生态里若有 DeFi 协议用验证者集合做 slashing 或委托逻辑，这类合约的兼容性谁来审查、怎么审查？提案没指定责任方。
2. **"更大步长风险高"缺数据**：这是定性判断，原文没有给出 300 验证者下的性能测试或模拟，来证明 +100 是安全边界而非经验拍脑袋。
3. **Draft 状态无量化模拟**：尚无 300 验证者下的实测 TPS / 最终性延迟数据，结论目前停留在"论证合理"层面，未到"数据验证"层面。

### 潜在反对意见（推测）
- **"现在就需要吗？"**：如果当前 200 个位置远未满，提高上限是未雨绸缪而非解决痛点，可能被认为优先级低。
- **"为什么不是参数化？"**：与其每次改数字都走 MIP + 硬分叉，为何不把活跃集合大小做成治理可调参数？这反映协议成熟度问题。

## 产品启发

1. **去中心化是产品卖点，MIP 是落地证据**：Monad 对外强调"高性能不牺牲去中心化"，MIP-9 是这句话的工程落地——用 MonadBFT 的架构优势（线性通信）支撑更大的验证者集合。作为 Research，看一个项目"说什么"和"MIP 里做什么"是否一致，比看营销文案可靠。

2. **治理透明度是可观察信号**：MIP 论坛公开了提案全文、讨论、投票节奏。研究者可以用它判断项目方是否在认真做去中心化治理，还是中心化团队说了算。这是评估公链长期价值的一手信息源。

3. **参数化思维反映协议成熟度**：好的协议设计会把"治理可调参数"和"硬编码常量"分开。MIP-9 暴露了 Monad 目前活跃集合还是硬编码（需硬分叉改），说明协议仍处于早期调参阶段，治理自动化程度有限。

4. **"小步快跑"的参数治理模式**：+100 而非激进扩容，体现了工程上的谨慎。对产品团队来说，这种"可验证的渐进式变更"比"一次性大改"风险更可控，值得在自家产品迭代中借鉴。

## 我的疑问（研究视角）

1. **MIP-9 是"供给侧扩容"还是"需求侧压力"？** 如果当前 200 位置远未满，这更像是主动为未来增长铺路，而非解决当下瓶颈。判断一个 MIP 是"引领"还是"补救"，能区分团队节奏。

2. **Monad "去中心化"叙事和参数现实差距多大？** 官网强调消费级硬件 + 大验证者集合，但活跃集合仍靠硬编码上限管控，且 Core 改动需硬分叉——说明"去中心化"目前还不是链上治理自动调节的。

3. **"改一个数字"类 MIP 在 Monad 治理中占比多少？** 如果大量 MIP 都是 Core 硬编码参数调整，说明协议成熟度还早，治理更多在"调参"而非"架构演进"——这对判断生态所处阶段有信号意义。

4. **MIP 论坛的提案通过率如何？** Draft 状态的 MIP 有多少最终被激活？通过率高低反映治理是实质运作还是形式讨论。

## 一句话总结

> MIP-9 把一个数字从 200 改成 300，背后是 Monad"高性能不牺牲去中心化"路线的具体体现——但作为 Draft，它还需要性能数据证明 +100 是安全边界，也折射出早期公链治理仍以"核心团队提案 + 参数硬编码"为主的特征。对 Research 而言，这类 MIP 本身就是观察项目成熟度的一手样本。
