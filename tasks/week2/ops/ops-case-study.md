# Ops Case Study：一场"不写合约也能接协议"的 Space 实战夜

## 活动简介

**「不写一行合约，也能让 AI 帮你质押：用 Moss 在 Monad 上接一个协议」**——一场面向 Web3 新人的 60 分钟 Twitter Space。

主题来自本周真实落地的技术工作：把 aPriori 的 MON 流动性质押封装成 Moss 的 AI Agent adapter（PR #104）。运营上要做的事，是把这个偏技术的案例，翻译成新人"听得懂、想参与"的内容，并设计一套能直接上线的活动与物料。

## 本周运营成果

围绕这一主题，本周完成了从策划到物料的完整链路：

| 产出 | 内容 |
|------|------|
| 活动策划案 | 主题、目标、目标听众、核心问题、举办原因、参考案例（Monad Builder Camp / Moss / aPriori） |
| 内容设计（Run of Show） | 60 分钟 7 段议程、5 个核心讨论问题、"一句话需求→猜函数"互动、三级收尾 CTA |
| 执行预案 | 3 周倒推排期、宣传节奏、6 人分工、当天 9 步流程、6 类风险预案、预期目标 |
| 运营物料 | 完整宣传文案、简短报名页、开场前双时段提醒、收尾 CTA（star / fork / 认领任务） |

所有内容都锚定同一个真实案例（aPriori adapter PR #104），未做虚饰，活动本身未实际举办（按任务要求只做设计与物料）。

## 核心运营思考（最重要的一个判断）

**最重要的运营判断：用"真实踩坑案例"替代"完美教程"。**

一开始很容易把活动做成"Moss 框架功能讲解"——但那对新人是噪音。我选择把 aPriori adapter 从"假设 stake() 单参"到链上确认 EIP-7702 委托合约、再到按官方文档补全三步的**真实踩坑过程**作为主线。原因是：

- 新人最大的门槛不是"不懂概念"，而是"不敢动手"——看到别人也踩坑、也靠自查解决，心理门槛立刻降下来。
- 案例已落地（PR #104 链上验证），所有素材可核对，不是演示稿，可信度高。
- 一条"MVP 单 Capability → 完善多步骤 + 链上验证"的方法论能从这场活动抽出来，套到 Kuru、其他 LST、DEX，活动价值不随单个协议过期。

配套的设计原则是"低门槛 CTA"：star（一键）→ fork 跑通（中等）→ 认领任务（高），让不同意愿的听众都有下一步，而不是听完即散。

## AI 协作记录

**AI 协助的部分：**
- 基于 Moss 框架与 aPriori 文档，快速生成 adapter 初始代码骨架（单 `stake()` Capability），以及活动策划/Run of Show/执行预案/物料的初稿结构与文案。
- 把零散的技术发现（EIP-7702、selector、e2e 输出）整理成可读的文档与流程图。

**我自己修改和完善的部分（关键判断在人）：**
- **技术侧**：发现 AI 初始骨架按"标准 ERC4626 stake()"假设，主网 simulate revert。我链上 decode bytecode 确认 aprMON 是 EIP-7702 委托合约，最终按 aPriori 官方文档改为 payable `deposit(uint256,address)` + 异步 `requestRedeem`/`redeem`，并补链上验证注释（ADR 0007）。这一步 AI 给的骨架是错的，方向是我纠正的。
- **运营侧**：AI 初稿偏"功能罗列"，我把它重构为"踩坑案例主线 + 低门槛 CTA"的叙事；互动环节从泛泛讨论改成"猜函数"现场接龙，让听众当场练"读文档→写调用"；风险预案按小场实际（冷场、嘉宾缺席）补了可操作应对，而非模板套话。
- **任务格式**：按平台要求删去文档里的学分/赛道/任务描述性元数据，只留内容本身。

一句话：AI 负责"快出骨架与初稿"，人负责"纠方向、定叙事、补真实证据"。
